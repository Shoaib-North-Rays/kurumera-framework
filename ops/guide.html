K KURUMERA · THEME FRAMEWORK
Developer guide

Build storefronts as code, publish them per‑store.

Kurumera lets you write a real Next.js theme, run it against a live store's products and cart, and ship it to a merchant's storefront with one command — Shopify-style themes, but React all the way down.

CLI @kurumera/cli Runtime Next.js 15 Node 20+ Storefront API reference → Theme marketplace →
01
Concept

What you're working with

A theme is a Next.js app you author locally. It renders a store's home, product, collection, cart and search pages, pulling live data from the Storefront API through a read-only token — so what you see in dev is the merchant's real catalog, not fixtures.

You never run servers or manage hosting. The kurumera CLI uploads your source; the platform builds it in a sandbox, hosts a versioned container per store, and serves it at <store>.kurumera.com. Every version, publish, and rollback is recorded, so a store can always go back.

You write

The theme

React components + templates against a fixed route contract. Client-safe only.

The CLI does

Build & publish

check → push → preview → publish → rollback, scoped to stores you own.

The platform runs

Hosting

Sandboxed builds, versioned per-store containers, history in Postgres.

03
From scratch

The developer journey

Nine commands, start to shipped. Run them in order the first time; after that you'll live in dev → push → publish.

1

Install the CLI

One global install. Requires Node 20 or newer.

bash
$ npm i -g @kurumera/cli
$ kurumera --help
2

Sign in

Opens your browser to authorize, then hands the session back to the CLI. It remembers your default store so most commands need no --store.

bash
$ kurumera login
Opening your browser to sign in…
✓ Logged in — store: your-store
3

Scaffold a theme

Creates a complete, valid theme — every required route, the config, and the storefront SDK wired up.

bash
$ kurumera theme init my-theme
$ cd my-theme
4

Develop against live data

Runs the theme locally, resolving your store so components render the real catalog. Edit and hot-reload as usual.

bash
$ kurumera theme dev
▲ Local:  http://localhost:3000  ·  store: your-store
5

Validate

Checks the route contract and safety rules before anything leaves your machine. Grouped output with a fix hint on each finding. Push is blocked until this passes.

bash
$ kurumera theme check
✓ theme check passed — no issues.  Ready to push.
6

Push — the platform builds it

Bundles your source and uploads it. The platform builds it in an isolated sandbox and stages a preview. You can only push to a store you staff.

bash
$ kurumera theme push
▸ Uploading theme (20 KB)…
✓ Pushed — version queued. Building…
$ kurumera theme logs   # watch the build
7

Preview

Opens the built version on a hosted, store-scoped URL — the real thing, not yet live to shoppers.

bash
$ kurumera theme preview
→ https://themekit.kurumera.com/?store=your-store
8

Publish — go live

Makes the built version the store's live theme. Shoppers on your-store.kurumera.com now see it. --off reverts to the visual builder.

bash
$ kurumera theme publish
✓ Published — live at https://your-store.kurumera.com
9

Roll back, anytime

Restores the store's previous live version instantly. Every publish is on record, so this is always safe.

bash
$ kurumera theme rollback
✓ Rolled back to the previous version.
04
Structure

Anatomy of a theme

A theme is an ordinary Next.js App Router project with a fixed set of routes. Highlighted paths are required — theme check enforces them.

my-theme/
├─ theme.config.ts        # name, version, routes, merchant settings
├─ middleware.ts          # resolves which store is being served
├─ app/
│  ├─ page.tsx             # home
│  ├─ products/[handle]/page.tsx
│  ├─ collections/[handle]/page.tsx
│  ├─ cart/page.tsx
│  ├─ search/page.tsx
│  ├─ pages/[handle]/page.tsx
│  └─ not-found.tsx       # branded 404 (recommended)
└─ package.json           # depends on @kurumera/storefront
The contract

Data via the SDK

Import @kurumera/storefront to read products, collections, cart and search. It resolves the current store for you — no keys in the theme.

The rules

Client-safe only

No fs/child_process, no app/api, no "use server", no server-only deps. Enforced locally by check and re-checked on the server.

05
Your stores

Seeing & managing your stores

You act on stores you're staff of — the platform verifies ownership on every push and publish, so a store header alone can't reach a store you don't run.

From the CLI

Your default store

kurumera login prints it and every command uses it unless you pass --store <slug>. Your storefront is always <slug>.kurumera.com.

From the dashboard

kurumera.com / themes

See the current storefront (visual builder vs code theme), the live & preview versions, the full history timeline, and publish, roll back, or install a theme — no CLI needed.

Multi-store

Own more than one store? Each has its own --store slug. Target any of them explicitly, e.g. kurumera theme publish --store north-cafe. Trying to touch a store you don't staff returns 403 — you do not have access to this store.

06
Live data locally

Reading real store data in dev

kurumera theme dev uses your login to resolve the store, so the SDK returns live products and collections while you build. For off-host or headless use (CI, a preview box, a separate frontend), mint a read-only storefront token.

  • Mint it in the dashboard → Themes → Storefront tokens. It's shown once and starts with ksf_.
  • Scope — anonymous, read-only, tenant-locked. It can only reach public storefront reads for that one store; it can never write or reach an authenticated endpoint.
  • Use it — set KURUMERA_STOREFRONT_TOKEN in the environment, or send it as a Bearer ksf_… header from a headless client.

08
Distribution

The marketplace

Build a theme once, install it into many stores. marketplace publish promotes a store's built version into a shared registry as an immutable theme@version. Anyone can then install it into a store they own — the artifact is copied into that store's own history, so installs stay isolated and roll back independently.

bash
# publish the build you just pushed
$ kurumera marketplace publish
✓ Published aurora@1.0.0 to the marketplace.

# install it into another store you own
$ kurumera marketplace install aurora --store north-cafe
✓ Installed aurora@1.0.0 into "north-cafe" — it's now live.
09
Guarantees

What the platform handles for you

Security

Sandboxed builds

Your build and runtime run unprivileged, capabilities dropped, on a read-only filesystem. Builds run on an isolated network with no route to internal services.

Isolation

Per-store versioning

Each store gets its own versioned build and container. One store's theme can never affect another; every version is immutable once built.

Efficiency

Scale-to-zero

Idle stores are paused and woken on the next request (~a few seconds cold), so one host serves far more stores than it could keep hot.

Durability

Self-healing state

Publish state lives in Postgres and reconciles continuously — a crashed container is restarted and the record always reflects what's actually running.

10
Checkout

Checkout — handled, and yours to style

Your theme never builds checkout. The cart's Checkout button (checkoutHref() from @kurumera/storefront) hands off to the platform's shared, secure checkout on checkout.<store>.kurumera.com — rendered with the store's brand theme, so it matches your storefront automatically. There's no checkout route to implement, and theme check doesn't require one.

Handled for you

No checkout to build

Your theme ships the cart; checkout and payments are platform-hosted and PCI-safe. You never touch payment code.

On brand

Inherits the theme

The store's colours and fonts flow into checkout, so it looks like the rest of the site out of the box.

Merchant-editable

Styled in the builder

On the Checkout page a merchant restyles it, renames field labels, and shows/hides optional fields — last name, apartment, postal code, country, phone.

Reliable orders

Fixed field set

Merchants relabel and toggle the standard fields, but can't add custom ones — so every order stays complete, valid, and fulfillable.

For merchants

To customise it: open the Checkout page in the builder → select the checkout form → change colours, labels, and which optional fields show in the inspector → Publish. About two minutes, no code. Full walkthrough →

11
If something's off

Common answers

  • “theme check failed” — read the grouped output; each finding has a → fix hint. Push stays blocked until it's clean.
  • “Push rejected (403)” — the target store isn't one you staff. Use --store with a store you own, or kurumera login as the right account.
  • First load after idle is slow — expected. An idle store cold-starts in a few seconds, then stays warm.
  • Build failed — run kurumera theme logs to see the exact npm install / next build output.
  • Need to undo a publish — theme rollback, or theme publish --off to return to the visual builder.
